# 사용자 이름(users.name) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** `users` 테이블에 표시용 이름을 추가하고, 상단바와 관리자 사용자 목록이 이메일 대신 이름을 보여준다.

**Architecture:** DB에 `name` NOT NULL 컬럼을 추가하되 기존 행은 이메일의 `@` 앞부분으로 백필한다. 회원가입 API가 이름을 받아 1~50자로 검증하고, 로그인·`/auth/me` 응답에 이름을 실어 프론트가 상단바에 표시한다. 관리자 목록은 라우터가 행을 그대로 내보내므로 SELECT에 컬럼만 추가하면 자동으로 반영된다.

**Tech Stack:** PostgreSQL · Alembic · SQLModel · FastAPI/Pydantic · aiosql · React 19 + TypeScript

**상위 스펙:** [docs/superpowers/specs/2026-07-24-user-name-design.md](../specs/2026-07-24-user-name-design.md)

## Global Constraints

- **DB의 `users.name`은 NOT NULL이다.** 마이그레이션은 `NULL 추가 → 백필 → NOT NULL 승격` 3단계여야 기존 행이 있어도 실패하지 않는다.
- **백필값은 `split_part(email, '@', 1)`** — `dev@bluenmobile.com` → `dev`.
- **`User` 모델 필드는 `default=""`** 를 갖는다. 테스트 18개 파일이 `User(...)`를 29번 생성하는데 기본값이 없으면 전부 깨지기 때문이다. 프로덕션 쓰기 경로는 모델이 아니라 raw SQL `insert_user`라 안전하다.
- **이름 검증: `strip()` 후 1~50자.** 위반 시 `AppError(400, "INVALID_NAME", "이름은 1~50자로 입력해 주세요.")`.
- **유니크 제약 없음** — 동명이인 허용. 식별자는 여전히 email.
- **이름에는 `lower()`를 적용하지 않는다** (email과 다르다).
- **컬럼 코멘트:** `"표시용 이름 (로그인 식별자는 email)"` — 기존 모델의 코멘트 관례를 따른다.
- 사용자 노출 메시지는 한국어. 파이썬 명령은 `uv run`, 프론트 명령은 `web/`에서 실행한다.
- 프론트에는 테스트 러너가 없다. 검증은 `npm run build`(tsc) + `npm run lint`.

`default=""`는 테스트가 `User(...)`를 이름 없이 만들 수 있게 하는 장치다. DB 제약(NOT NULL)과 충돌하지 않는다 — 빈 문자열은 NULL이 아니다.

- [ ] **Step 4: 마이그레이션 생성**

Run: `uv run alembic revision -m "add user name"`

생성된 파일의 `upgrade`/`downgrade`를 아래로 **교체**한다. autogenerate를 쓰지 않는 이유: 백필 UPDATE는 자동 생성되지 않고, 순서(추가 → 백필 → 승격)가 이 마이그레이션의 핵심이다.

```python
def upgrade() -> None:
    """Upgrade schema."""
    # 기존 행이 있으면 NOT NULL로 바로 추가할 수 없다. 3단계로 나눈다.
    op.add_column(
        'users',
        sa.Column('name', sqlmodel.sql.sqltypes.AutoString(), nullable=True,
                  comment='표시용 이름 (로그인 식별자는 email)'),
    )
    # 백필: dev@bluenmobile.com -> dev. 이메일 전체보다 이름답고 나중에 고치기 쉽다.
    op.execute("UPDATE users SET name = split_part(email, '@', 1)")
    op.alter_column('users', 'name', nullable=False)


def downgrade() -> None:
    """Downgrade schema."""
    op.drop_column('users', 'name')
```

파일 상단 import에 `import sqlmodel`이 있는지 확인한다(기존 마이그레이션들과 동일). 없으면 추가한다.

// 클라이언트 검증은 UX 보조일 뿐 신뢰 경계가 아니다. 진짜 검증은 서버가 한다.
function validate(name: string, email: string, password: string, confirm: string): FieldErrors {
  const errors: FieldErrors = {}
  const trimmed = name.trim()
  if (trimmed.length < 1 || trimmed.length > 50) {
    errors.name = '이름은 1~50자로 입력해 주세요.'
  }
  if (!/^[^\s@]+@[^\s@]+\.[^\s@]+$/.test(email)) {
    errors.email = '올바른 이메일 형식이 아닙니다.'
  }
  if (password.length < 8) {
    errors.password = '비밀번호는 8자 이상이어야 합니다.'
  }
  if (password !== confirm) {
    errors.confirm = '비밀번호가 일치하지 않습니다.'
  }
  return errors
}
