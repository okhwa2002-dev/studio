# JWT 비밀키 강제 검증 설계

## 목적

HS256 서명에 사용하는 `JWT_SECRET`이 권장 최소 길이보다 짧을 때 애플리케이션이
기동되지 않도록 한다. 테스트에서도 충분히 긴 키를 사용해 PyJWT의
`InsecureKeyLengthWarning`을 제거한다.

## 범위

- `Settings.jwt_secret`을 UTF-8 인코딩 기준 최소 32바이트로 검증한다.
- 32바이트 미만이면 Pydantic 설정 검증 오류를 발생시킨다.
- 정확히 32바이트인 값과 그보다 긴 값은 허용한다.
- 테스트와 `.env.example`에서 사용하는 JWT 비밀키를 32바이트 이상으로 교체한다.
- 실제 `.env`는 비밀 파일이므로 자동으로 수정하지 않는다.

JWT 발급 형식, access/refresh token 만료 시간, 쿠키 설정, 서명 알고리즘은 변경하지
않는다.

## 구현 방식

`app/config.py`의 `Settings`에 `jwt_secret` 필드 검증기를 둔다. Python 문자열의
글자 수가 아니라 `len(value.encode("utf-8"))`를 검사한다. 이는 비ASCII 문자가
포함된 비밀키도 실제 서명 키의 바이트 길이로 판단하기 위해서다.

검증 실패 메시지는 설정 이름과 최소 길이 요구사항을 명확히 알리되, 비밀키 값은
포함하지 않는다. 설정 객체 생성 시 검증되므로 API 서버, Alembic, 관리 스크립트 등
`Settings`를 사용하는 모든 진입점에 동일하게 적용된다.

## 테스트

TDD 순서로 다음 동작을 검증한다.

1. 31바이트 비밀키로 `Settings`를 만들면 검증 오류가 발생한다.
2. 정확히 32바이트 비밀키는 허용된다.
3. 멀티바이트 문자는 UTF-8 바이트 길이로 판정된다.
4. 기존 테스트 fixture와 직접 생성하는 `Settings`의 키를 안전한 길이로 교체한다.
5. 전체 pytest를 실행해 모든 테스트가 통과하고 JWT 키 길이 경고가 사라졌는지
   확인한다.
6. 프런트 lint와 build를 실행해 저장소 전체 검증 상태를 확인한다.

## 오류 처리

잘못된 설정은 Pydantic `ValidationError`로 즉시 드러난다. 오류 메시지에는
`JWT_SECRET must be at least 32 bytes`를 포함한다. 서버가 약한 키로 부분 기동하거나
JWT 발급 시점까지 오류를 미루지 않는다.
